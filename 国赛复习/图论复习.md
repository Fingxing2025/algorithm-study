2. 连通性怎么维护。
3. DAG 上的先后关系怎么处理。
4. 最短路怎么按数据范围选算法。

- 看不出题目本质是图。
- 认出了是图，但选错了模型。
- 模型选对了，存图、初始化、边界条件写炸了。

## 1. 图论的本质

图论题本质上都在回答下面几类问题：

1. 两个点是否连通。
2. 点与点之间怎么走最优。
3. 所有点之间的依赖顺序是什么。
4. 如何用尽量小的代价把点连起来。
5. 在树上，两个点的关系如何快速定位。

所以做题时，先不要急着想模板，先判断这道题在问哪一类：

- 问“是不是一个集合”时，常常是并查集。
- 问“最短多少 / 最小多少”时，常常是最短路或最小生成树。
- 问“先做谁后做谁”时，常常是拓扑排序。
- 问“树上两点公共祖先 / 距离 / 路径关系”时，常常先想到 LCA。

## 2.5 蓝桥杯国赛视角下的优先级重排

如果按“图论通识”来学，这八个模板都应该掌握。

但如果按“蓝桥杯国赛一个多月冲刺”来排优先级，就不能平均发力，而要改成下面这个顺序：

### 第一优先级：必须稳定拿分

1. 链式前向星
2. 并查集
3. Dijkstra
4. 拓扑排序
5. Kruskal

原因：

- 这几类最容易直接落在国赛前中段题上。
- 它们兼顾实现题、图基础题、构图题和中档最优题。
- 练熟之后，不仅能做原题，还能支撑状态图、DAG DP、最短路建模这些变形。

### 第二优先级：要会判断，会写最小模板

1. Prim
2. Floyd
3. LCA

原因：

- Prim 和 Floyd 本身不一定高频，但你要知道什么时候该用、什么时候不该用。
- LCA 在通用竞赛里很基础，但在蓝桥杯近几年国赛里，优先级通常略低于并查集、最短路、拓扑和 MST。

### 第三优先级：当前图论体系还没补的块

如果你后面继续补图论模板，蓝桥杯视角下更推荐优先补：

1. BFS 状态图最短路
2. Tarjan 割边 / 割点
3. DAG 上路径计数 / 最长路

原因：

- 这些比更重的树链剖分、网络流，更容易在蓝桥杯国赛一个月窗口里转化成真实分数。

## 3. 存图专题：链式前向星

### 3.1 什么时候用它

当题目是：

- 点很多，边也很多。
- 需要频繁遍历某个点的所有出边。
- 想用数组而不是 `vector` 存图。

就很适合链式前向星。

### 3.2 这份模板在干什么

定义：

- `head[u]`：点 `u` 的第一条边编号。
- `edge[i].to`：第 `i` 条边指向谁。
- `edge[i].next`：与第 `i` 条边同起点的下一条边编号。

加边时：

```text
新边插到链表最前面
```

遍历点 `u` 的所有出边时：

```cpp
for (int i = head[u]; i != 0; i = edge[i].next) {
    int v = edge[i].to;
}
```

### 3.3 复习重点

1. 记住 `head` 存的是边编号，不是邻点编号。
2. 记住 `next` 是“同起点上一条边”的编号。
3. 无向图要加两次边。
4. `cnt` 一般从 `1` 开始更顺手，因为 `0` 可以表示链表结束。

### 3.4 易错点

- `N` 和 `M` 开太小。
- 无向图只加了一次边。
- 忘记清空 `head` 和 `cnt`。
- 遍历时把 `edge[i].next` 写错成 `head[i]`。

一句话总结：

> 链式前向星不是算法本身，而是很多图论模板的地基。

## 4. 连通性专题：并查集

### 4.1 它解决什么问题

并查集最适合处理：

- 两个点是否属于同一个连通块。
- 动态合并两个集合。
- 若干边加入后，图被分成几个块。

典型信号词：

- 连通块
- 合并
- 同组
- 家族 / 帮派 / 网络

### 4.2 这份模板的核心

- `fa[x]`：`x` 的父亲。
- `find(x)`：找根，并做路径压缩。
- `unite(x, y)`：把两个集合合并。

这份模板额外做了：

- `sz[]` 维护集合大小。
- 按大小合并，把小集合挂到大集合上。

### 4.3 为什么能快

因为有两个优化：

1. 路径压缩：查找时顺便把链拉平。
2. 按大小合并：尽量避免树退化太深。

### 4.4 易错点

- 合并时直接 `fa[x] = y`，没先找根。
- 按大小合并时大小数组更新错对象。
- 多组数据忘记 `init(n)`。
- 把并查集误用到“删边”“最短路”这种不适合的题。

一句话总结：

> 并查集擅长回答“是不是一家人”，不擅长回答“怎么走最优”。

## 5. DAG 顺序专题：拓扑排序

### 5.1 什么题该想到拓扑

当题目出现：

- 先修课程
- 任务依赖
- 必须先做 A 才能做 B
- 给定若干有向边，判断有没有环

就应优先想到拓扑排序。

### 5.2 这份模板的思路

Kahn 算法流程：

1. 统计每个点入度。
2. 把所有入度为 `0` 的点进队。
3. 每次取出一个点，删掉它的出边。
4. 某个点入度减到 `0` 时，再进队。

若最后取出的点数小于 `n`，说明图里有环。

### 5.3 复习重点

1. 拓扑排序只适用于 DAG。
2. 它不仅能输出顺序，还能判环。
3. 拓扑序常常是 DAG DP 的前置步骤。

### 5.4 易错点

- 有向图边方向建反。
- `in_degree` 被改坏后又拿去复用。
- 没判断结果长度是否等于 `n`。
- 题目要字典序最小拓扑序时，普通队列不够，要换优先队列。

一句话总结：

> 拓扑排序处理的是“先后关系”，不是“路径最优”。

## 6. 最短路专题

这组模板里有两种最短路：

1. Dijkstra：单源最短路。
2. Floyd：多源最短路。

它们最大的区别不是写法，而是适用范围。

### 6.1 Dijkstra：非负边权单源最短路

这份模板用的是：

- 链式前向星存图。
- 小根堆维护当前最短路估计值。

状态：

- `dist[v]`：起点到 `v` 的当前最短距离。

流程：

1. 起点距离设为 `0`。
2. 每次从堆里取当前距离最小的点。
3. 用这个点去松弛它的所有出边。

#### Dijkstra 什么时候用

- 边权非负。
- 只关心一个起点到其他点。
- 图比较稀疏。

#### Dijkstra 易错点

- 有负边还硬上 Dijkstra。
- 堆里弹出的旧状态没判掉。
- `INF` 开太小。
- 无向图忘记双向加边。

一句话总结：

> Dijkstra 解决的是“从一个起点出发，怎么最快到别处”。

### 6.2 Floyd：任意两点最短路

这份模板用的是邻接矩阵：

- `f[i][j]` 表示 `i` 到 `j` 的最短距离。

核心转移：

```text
f[i][j] = min(f[i][j], f[i][k] + f[k][j])
```

这里的关键语义是：

- 枚举中转点 `k`，看经过 `k` 会不会更优。

#### 什么时候用

- 点数较小。
- 需要查询很多对点之间的最短路。
- 想一步把全源最短路都预处理出来。

#### Floyd 易错点

- 三层循环顺序写错，`k` 必须在最外层。
- 邻接矩阵初始化没设 `f[i][i] = 0`。
- 没判 `INF` 就直接相加，导致溢出。
- 点数很大还用 Floyd，直接超时。

一句话总结：

> Floyd 适合“小图多询问”，不适合“大图单源”。

### 6.3 Dijkstra 和 Floyd 怎么选

| 情况                             | 优先考虑               |
| -------------------------------- | ---------------------- |
| 单源最短路，边权非负，图较大     | Dijkstra               |
| 多次查询任意两点最短路，点数较小 | Floyd                  |
| 图很稠密、点不大                 | Floyd 也可能更直接     |
| 有负边                           | 这两份模板都不直接适用 |

## 7. 最小生成树专题

这组模板里有两种最小生成树：

1. Kruskal：按边选。
2. Prim：按点扩展。

它们都在解决同一件事：

- 用最小总代价把所有点连成一棵树。

### 7.1 Kruskal：按边权排序后贪心选边

这份模板流程：

1. 把所有边按边权从小到大排序。
2. 从小到大枚举边。
3. 若两端点不连通，就把这条边选进生成树。

为什么要配并查集：

- 因为要快速判断“这条边会不会形成环”。

#### 易错点

- 排序后没有判断两端是否已连通。
- 生成树边数不是 `n - 1` 时忘记输出无解。
- 无向图边读入混乱。

一句话总结：

> Kruskal 的思路是“从小边开始，能要就要，但不能成环”。

### 7.2 Prim：从一个点开始不断扩展

这份模板是邻接矩阵版 Prim。

状态：

- `dis[j]`：点 `j` 到当前生成树的最小连边代价。

流程：

1. 任选一个点进入生成树。
2. 每次选择离当前生成树最近的点加入。
3. 用新加入的点更新其他点的最小连边代价。

#### Prim 什么时候用

- 稠密图。
- 邻接矩阵写起来更直接时。

#### Prim 易错点

- 最小生成树是无向图，建图只建单向边。
- 重边没有保留较小边权。
- 图不连通时没判无解。
- `u == -1` 或 `dis[u] == INF` 的情况没处理。

一句话总结：

> Prim 的思路是“每次把离生成树最近的点拉进来”。

### 7.3 Kruskal 和 Prim 怎么选

| 情况                         | 优先考虑              |
| ---------------------------- | --------------------- |
| 边集天然给出、边数不算太夸张 | Kruskal               |
| 图较稠密、邻接矩阵更方便     | Prim                  |
| 已经有并查集模板、想写得稳   | Kruskal               |
| 点多边多但矩阵存不下         | 不要用这份矩阵版 Prim |

## 8. 树上专题：LCA

### 8.1 它解决什么问题

LCA 是最近公共祖先。

典型问题：

- 两个点的最近公共祖先是谁。
- 树上两点距离。
- 两点路径是否经过某个点。
- 树上路径信息拆分。

### 8.2 这份模板的核心思路

这份模板是倍增版 LCA。

预处理：

- `depth[u]`：点 `u` 的深度。
- `fa[u][i]`：点 `u` 的 `2^i` 级祖先。

查询时分两步：

1. 先把更深的点跳到同一深度。
2. 再从大到小同时往上跳，直到它们父亲相同。

### 8.3 为什么是倍增

因为每次不是只跳一步，而是尽量跳 `2^i` 步。

这样单次查询复杂度是：

- `O(log n)`

### 8.4 易错点

- 树根没定清楚。
- `dfs` 时父子关系没判好。
- `fa[u][i]` 的递推下标写错。
- 深度提升时没先保证 `x` 是更深的那个。
- 把 LCA 用在一般图上。

一句话总结：

> LCA 是树上路径问题的入口模板，不是普通图模板。

## 9. 这八个模板应该怎么串起来复习

推荐顺序：

### 第一层：图的基础表达

先熟：

- 链式前向星
- 邻接表 / 邻接矩阵的使用场景

要求：

- 能独立写出存图。
- 知道无向图要加双边。

### 第二层：最基础的结构题

再练：

- 并查集
- 拓扑排序

这两类题很适合建立“图不一定是在跑最短路”的意识。

### 第三层：图上最优问题

然后练：

- Dijkstra
- Floyd
- Kruskal
- Prim

这里最重要的不是背代码，而是建立判断：

- 单源还是多源。
- 稀疏还是稠密。
- 求最短路还是求最小生成树。

### 第四层：树上查询

最后练：

- LCA

因为它既用到 DFS，也用到倍增，属于“图 + 树 + 预处理”的综合模板。

如果改成蓝桥杯国赛冲刺顺序，更推荐：

1. 链式前向星
2. 并查集
3. Dijkstra
4. 拓扑排序
5. Kruskal
6. Prim / Floyd 只做模型辨认
7. LCA 作为树上基础补齐

## 10. 图论最常见误区

### 10.1 见图就最短路

很多题虽然给了边，但实际在问：

- 连通性
- 是否有环
- 依赖顺序
- 最小生成结构

这时不该先想 Dijkstra。

### 10.2 见树就 LCA

有些树题只需要：

- DFS 一遍
- 记录父亲
- 简单递推

不用一上来就倍增。

### 10.3 最小生成树和最短路混淆

要分清：

- 最短路：让一条路径最优。
- 最小生成树：让整张图联通总代价最优。

这两个目标完全不同。

### 10.4 忘记图是否有向

这是最常见实现错误之一：

- 拓扑排序是有向图。
- 最小生成树是无向图。
- 最短路两种都可能。

### 10.5 忘记数据范围

比如：

- 点数大还用 Floyd。
- 边很多却数组开不够。
- 距离和边权用 `int` 直接溢出。

## 11. 一页速记

如果把这组模板压缩成一页脑图，就是下面这几句：

1. 图先看是有向还是无向。
2. 先判断问的是连通、顺序、最短、生成还是树上关系。
3. 连通性先想并查集。
4. DAG 顺序先想拓扑。
5. 单源非负最短路先想 Dijkstra。
6. 小图全源最短路先想 Floyd。
7. 最小生成树会区分 Kruskal 和 Prim。
8. 树上祖先关系先想 LCA。
9. 链式前向星是很多图论模板的基础设施。

最后一句话总结整组模板：

> 图论复习的关键不是把模板背得很像，而是看到题后，能在几秒内判断“这题到底是在问图里的哪一种关系”。
